What it does

Overview & Capabilities

VectorRAG turns any local document into a conversational knowledge base. Upload once, chat naturally — the semantic engine finds the most relevant pages even when your question uses different words than the document.

🧠

Semantic Search

Uses TF-IDF with bigrams + TruncatedSVD (LSA) to create 128-dimensional dense vectors. Finds semantically similar pages even when exact keywords don't match.

📂

6 File Types

Supports PDF (pdfplumber), DOCX (python-docx), XLSX/XLS (openpyxl), CSV, TXT, and Markdown. Each parsed into natural page units — no arbitrary chunking.

⚡

Fully Local

No HuggingFace downloads, no cloud vector DB, no GPU. sklearn + numpy only. Works on air-gapped corporate networks.

💬

Multi-turn Chat

Maintains conversation history per session. Follow-up questions reference previous answers naturally.

🤖

18+ LLM Models

Switch between GitHub Copilot models (GPT-5, Claude Opus/Sonnet/Haiku, Grok) at any time from the sidebar.

📖

Document Browser

Browse all indexed pages with search, stats, and full-content preview. Semantic search ranks pages by cosine similarity.

Setup

Getting Started

Get VectorRAG running in three steps. No API keys to register, no model downloads — just pip install and start the server.

Step 1 — Install Dependencies

Install all Python packages from the requirements file.

cd vector-rag-app-v1 pip install -r requirements.txt

Step 2 — Start the Server

Launch the FastAPI server. It starts on port 8000 by default.

python server.py
✅ Server starts at http://localhost:8000 — open this in your browser.

Step 3 — Upload a Document & Chat!

Open the app in your browser, upload any supported document, and start asking questions.

Upload & Chat Flow

  1. Enter your GitHub token — or let it auto-detect from gh auth token.
  2. Select Provider (GitHub Models or GitHub Copilot) and Model from the sidebar.
  3. Set Top-K — number of pages to retrieve per query (default: 4).
  4. Drop or choose a file: PDF, DOCX, XLSX, CSV, TXT, or MD.
  5. Wait for indexing — TF-IDF vocabulary build + LSA vectorization (typically <2s for most documents).
  6. The Chat tab opens automatically — ask anything about your document!

Under the Hood

VectorIndex Deep Dive

The VectorIndex class (src/vector_index.py) is the heart of VectorRAG. It implements a four-stage pipeline: TF-IDF sparse vectorization → LSA dense reduction → L2 normalization → cosine similarity search.

A

TF-IDF Vectorization

  • max_features=15000 — vocabulary cap
  • stop_words="english" — removes common words
  • sublinear_tf=True — log normalization of term freq
  • ngram_range=(1,2) — unigrams + bigrams
  • Output: sparse matrix (n_pages, 15000)
B

LSA via TruncatedSVD

  • n_components=128 — latent dimensions
  • random_state=42 — reproducible runs
  • Captures latent semantic structure
  • Handles synonyms & related terms
  • Output: dense matrix (n_pages, 128)
C

L2 Normalization

  • normalize() from sklearn.preprocessing
  • Makes each row unit-length vector
  • Dot product of normalized vectors = cosine similarity
  • Eliminates document length bias
  • Output: embeddings matrix (n_pages, 128)
D

Query Search

  • Transform: TF-IDF → SVD → normalize (same pipeline)
  • scores = embeddings @ query_vec.T
  • Returns top-K by argsort descending
  • Score range: 0.0 to 1.0 (cosine similarity)
  • Sub-millisecond for most documents

Search Method Source

# src/vector_index.py — VectorIndex.search() def search(self, query: str, top_k: int = 4) -> List[Tuple[Page, float]]: # Transform query through same TF-IDF → SVD → normalize pipeline q_tfidf = self.vectorizer.transform([query]) # sparse (1, 15000) q_vec = normalize(self.svd.transform(q_tfidf)) # dense (1, 128) # Cosine similarity = dot product of L2-normalized vectors scores = (self.embeddings @ q_vec.T).flatten() # (n_pages,) # Return top-K pages sorted by similarity score descending top_idx = np.argsort(scores)[::-1][:min(top_k, len(self.pages))] return [(self.pages[i], float(scores[i])) for i in top_idx]
💡 Why LSA? Unlike raw TF-IDF, LSA captures latent semantic relationships between terms. If a document discusses "myocardial infarction" and a user asks about "heart attack", LSA can still find the relevant page because the terms co-occur in similar semantic contexts across the corpus.

Index Construction

# src/vector_index.py — VectorIndex.__init__() # Stage 1: Build TF-IDF sparse matrix self.vectorizer = TfidfVectorizer( max_features=15000, stop_words="english", sublinear_tf=True, ngram_range=(1, 2), ) tfidf = self.vectorizer.fit_transform(texts) # shape: (n_pages, 15000) # Stage 2: LSA — reduce to dense semantic vectors n = min(n_components, len(texts) - 1, tfidf.shape[1] - 1) self.svd = TruncatedSVD(n_components=max(n, 1), random_state=42) dense = self.svd.fit_transform(tfidf) # shape: (n_pages, 128) # Stage 3: L2-normalise so dot product == cosine similarity self.embeddings = normalize(dense) # shape: (n_pages, 128)
⚠️ Minimum corpus size: TruncatedSVD requires at least 2 documents (pages) and the number of components is clamped to min(128, n_pages-1, vocab_size-1). Very small documents (<3 pages) will use fewer latent dimensions, which may reduce semantic quality.
Document Support

Supported File Types

VectorRAG supports six document formats, each parsed by a dedicated strategy that produces natural semantic units — no arbitrary character/token chunking.

Extension Source Type Parse Strategy Page Unit Library
.pdf pdf_page Text + table extraction; first non-empty line as title One page per PDF page pdfplumber (PyPDF2 fallback)
.docx docx_section Heading-based section split (Heading 1/2/3, Title style) One section per heading python-docx
.xlsx / .xls xlsx_sheet Each sheet → markdown table with header row One page per sheet openpyxl
.csv csv_batch Rows batched 50 per page into markdown tables 50 rows per page Python csv module
.txt text_section Regex heading split; fallback: 800-word paragraph batches One section per heading or 800 words Built-in (re)
.md md_section Markdown heading split on ## / ### One section per heading Built-in (re)
📄 Natural units, not chunks: Each page is indexed as a complete natural unit. Tables, code blocks, and paragraphs are never split mid-way. This means an XLSX sheet with a 200-row table stays as one coherent page, and a PDF chapter with rich formatting is preserved in full.

Parser Architecture

# src/parser.py — parse_file() dispatcher dispatch = { ".pdf": _parse_pdf, # pdfplumber → page text + tables ".docx": _parse_docx, # python-docx → heading-based sections ".xlsx": _parse_xlsx, # openpyxl → one Page per sheet ".xls": _parse_xlsx, # same as xlsx ".csv": _parse_csv, # csv module → 50-row batches ".txt": lambda b: _split_text_sections(b.decode(), "text_section"), ".md": lambda b: _split_text_sections(b.decode(), "md_section"), } # Each Page dataclass carries: @dataclass class Page: page_num: int # 1-based index title: str # heading or "Page N" content: str # full text of this unit word_count: int # auto-computed if 0 source_type: str # "pdf_page" | "docx_section" | etc.
UI Feature

Document Browser

The Browse tab gives you full visibility into every indexed page — search semantically, inspect metadata, and expand to read the full content of any page.

📊

Stats Row

At the top of the browser: total pages, total words, average words per page, and the detected source type(s) — all computed from the in-memory index.

🔍

Semantic Search Box

Type any query — results are re-ranked by cosine similarity score via GET /api/pages?search=.... A 400ms debounce prevents excessive requests while typing.

🃏

Page Cards

Each card shows: page number, title, word count, source type badge, and similarity score (when searching). Click any card to expand it.

📋

Full Content Preview

Expanded cards display the complete page content. Markdown/XLSX/CSV pages render as HTML tables; PDF/DOCX/TXT shown in monospace pre-block.

🏷️

Metadata Chips

Each expanded page shows metadata chips: source type, word count, page number, and cosine similarity score (displayed as a percentage when searching).

⚡

Instant Re-rank

Clearing the search box instantly restores document-order view. All re-ranking is done server-side with the already-built index — no re-vectorization needed.

💡 Use the browser to validate retrieval quality: Enter your chat question in the browser search box to see exactly which pages would be retrieved and their similarity scores. If the wrong pages rank highly, try rephrasing your question with more specific terms.
Common Issues

Troubleshooting

Most issues with VectorRAG fall into one of these categories. Check server logs (server.log) for detailed error traces.

✅ SSL/TLS Errors When Downloading a Model

Some RAG apps require downloading sentence-transformer models which can fail behind corporate SSL-inspection proxies.

Already solved. VectorRAG uses sklearn/numpy only — no HuggingFace model downloads are ever required. The only outbound connection is the LLM API call, which uses Windows PowerShell's WinHTTP (native Windows certificate store) for maximum corporate proxy compatibility.
❌ "Session not found" Error

You receive a 404 with "Session not found. Please upload a document first." when chatting or browsing pages.

Your session expired — the server was restarted and all in-memory sessions were cleared. Re-upload your document. Sessions are stored in memory only; persistence across restarts is not implemented. Keep the server running while using the app.
❌ Upload Fails with 500 Error

The upload endpoint returns HTTP 500 and the document is not indexed.

Check these:
1. Verify the file extension is one of: .pdf .docx .xlsx .xls .csv .txt .md
2. Ensure the file is not password-protected or corrupted
3. Run gh auth status to verify your token is valid for auto-detection
4. Check server.log / server_err.log for the full Python traceback
⚠️ Poor Search Results / Irrelevant Pages Retrieved

The retrieved source pages don't match what you're looking for, even though the information is in the document.

Try these in order:
1. Increase Top-K in the sidebar (e.g., from 4 to 8) to retrieve more candidate pages
2. Rephrase with more descriptive words — LSA excels at longer concept queries
3. Use the Document Browser to directly search and verify which pages contain the answer
4. Very short documents (<5 pages) may have limited LSA quality — the SVD has less data to work with
❌ Browser Tab Shows "Failed to load pages"

The Document Browser tab cannot fetch page data.

The session may have expired or the server restarted. Re-upload your document to recreate the session. Also verify the server is running at http://localhost:8000.
⚠️ First Upload Is Slow

The first upload takes several seconds before indexing completes.

This is expected behavior. sklearn builds the TF-IDF vocabulary across all pages and then runs TruncatedSVD (LSA). For large documents (100+ pages), this typically takes 2–5 seconds. Subsequent chat queries are instant — the embeddings matrix is pre-computed and stored in memory.
❌ GitHub Token Not Detected

The app cannot auto-detect your token and shows an empty token field.

Run gh auth login in your terminal to authenticate the GitHub CLI. The app calls gh auth token at startup. Alternatively, paste your token manually in the sidebar. The token needs the models:read scope for GitHub Models and a Copilot subscription for GitHub Copilot endpoint.
