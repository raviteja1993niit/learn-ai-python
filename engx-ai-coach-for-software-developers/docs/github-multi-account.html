GitHub Multi-Account CLI Configuration

Manage personal & work GitHub accounts, Copilot profiles, SSH keys, and Git identities — all on one machine

gh CLI v2.40+ Windows & macOS & Linux Copilot Free Supported

1. Overview

The GitHub CLI (gh) has supported multiple authenticated accounts since v2.40 (Oct 2023). You can have a personal account and a work/enterprise account simultaneously — and switch between them with a single command.

~/.config/gh/hosts.yml ← stores all accounts │ ├── github.com │ ├── users: │ │ ├── personal-user (token, git_protocol: ssh) │ │ └── work-user (token, git_protocol: https) │ └── active_user: personal-user │ └── github.company.com ← GHES / enterprise └── users: └── corp-user (token, git_protocol: https)
ScenarioRecommended Approach
Switch globally for a sessiongh auth switch --user <name>
Per-project token isolation$env:GH_TOKEN / export GH_TOKEN=
Copilot Free API per accountgh auth token --user <name>
Git commits with correct identityPer-repo git config user.email
SSH push to both accountsSeparate SSH keys + ~/.ssh/config aliases

Install / Update gh CLI

# Windows (winget)
winget install --id GitHub.cli

# Windows (scoop)
scoop install gh

# macOS
brew install gh

# Ubuntu/Debian
sudo apt install gh

# Verify version (need 2.40+)
gh --version
✅ Minimum version: gh version 2.40.0 — multi-account support was added in this release. Run gh upgrade or reinstall if older.

3. Adding Accounts

3.1 Add Your First Account

If you haven't logged in yet, or want to start fresh:

gh auth login

Follow the interactive prompts:

1
Select GitHub.com (or Enterprise if applicable)
2
Select protocol: HTTPS (recommended for beginners) or SSH
3
Authenticate via browser or paste a Personal Access Token (PAT)
4
Done — your account is stored in ~/.config/gh/hosts.yml

3.2 Add a Second Account

Run gh auth login again — it detects the existing account and adds a new one alongside it:

# This will add a SECOND account, not replace the first
gh auth login --hostname github.com

When prompted, log in with the different GitHub username/token. Both accounts will coexist.

3.3 Add a GitHub Enterprise Server Account

# Replace with your company's GitHub Enterprise hostname
gh auth login --hostname github.mycompany.com

3.4 Add Account via PAT (Non-Interactive)

# Generate a PAT at: Settings > Developer settings > Personal access tokens
# Scopes needed: repo, read:org, gist, workflow, copilot

# Windows PowerShell
$env:GITHUB_TOKEN = "ghp_xxxxxxxxxxxxxxxxxxxx"
gh auth login --with-token <<< $env:GITHUB_TOKEN

# macOS/Linux
echo "ghp_xxxxxxxxxxxxxxxxxxxx" | gh auth login --with-token --hostname github.com

3.5 Verify All Accounts Are Registered

gh auth status

Expected output with two accounts:

github.com
  ✓ Logged in to github.com account personal-user (keyring)
  - Active account: true
  - Git operations protocol: ssh
  - Token: gho_****
  - Token scopes: 'gist', 'read:org', 'repo', 'workflow'

github.com
  ✓ Logged in to github.com account work-user (keyring)
  - Active account: false
  - Git operations protocol: https
  - Token: ghp_****

4. Switching Accounts

4.1 Global Switch (Persistent)

Changes the active account for all subsequent gh commands in any terminal:

# Interactive — shows a picker
gh auth switch

# Direct — specify username
gh auth switch --user work-user

# For enterprise hostname
gh auth switch --hostname github.mycompany.com --user corp-user

# Confirm switch worked
gh api user --jq .login

4.2 Per-Session Override (Environment Variable)

Does not change the global active account — only affects the current terminal session:

# Windows PowerShell
$env:GH_TOKEN = (gh auth token --user work-user)
gh api user --jq .login  # → work-user

# macOS/Linux
export GH_TOKEN=$(gh auth token --user work-user)
gh api user --jq .login  # → work-user

# Unset to go back to global active account
Remove-Item Env:GH_TOKEN  # PowerShell
unset GH_TOKEN            # bash
✅ Best practice for parallel work: Open two terminal windows. Set GH_TOKEN to the personal account token in one, and the work account token in the other. Both run simultaneously without interfering.

4.3 Per-Project (Directory-Scoped)

Use a .env file in the project root and load it when entering the directory:

# work-project/.env  (do NOT commit this file)
GH_TOKEN=ghp_workAccountTokenHere
# Load in PowerShell (add to profile or run manually)
Get-Content .\.env | ForEach-Object {
    if ($_ -match "^([^#][^=]*)=(.*)$") {
        [System.Environment]::SetEnvironmentVariable($matches[1], $matches[2], "Process")
    }
}

# Load in bash/zsh
export $(grep -v '^#' .env | xargs)
⚠️ Never commit .env — add it to .gitignore. Use .env.example with placeholder values for documentation.

5. GitHub Copilot CLI — Per Account

5.1 Verify Copilot Access per Account

# Check which account is active
gh api user --jq .login

# Check Copilot subscription status
gh api /copilot_internal/v2/token --jq .sku

# Switch to account with Copilot access
gh auth switch --user personal-user

# Confirm Copilot CLI extension works
gh copilot explain "what is a closure in JavaScript"

5.2 Copilot Free API Tokens per Account

The GitHub Copilot Free API (Azure AI Inference endpoint) uses the active account's token. Here's how to use both accounts' tokens in code:

import subprocess
from openai import OpenAI

# Get token for a specific account (even if not globally active)
def get_copilot_client(github_user: str) -> OpenAI:
    token = subprocess.run(
        ["gh", "auth", "token", "--user", github_user],
        capture_output=True, text=True
    ).stdout.strip()
    return OpenAI(
        base_url="https://models.inference.ai.azure.com",
        api_key=token
    )

# Use personal account (free tier — gpt-4o, claude-3.5-sonnet)
personal_client = get_copilot_client("personal-user")

# Use work account (may have Copilot Business/Enterprise)
work_client = get_copilot_client("work-user")

# Call with personal account
response = personal_client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello from personal account!"}]
)
print(response.choices[0].message.content)

6. Git Commit Identity per Account

Even with the correct gh account active, Git commits will use the global user.name / user.email unless overridden per repository.

6.1 Set Identity per Repository

# Inside a work project repo
git config user.name  "Raviteja Thota"
git config user.email "raviteja.thota@company.com"

# Inside a personal project repo
git config user.name  "Raviteja Thota"
git config user.email "raviteja1993@gmail.com"

# Verify
git config --list | Select-String "user"   # PowerShell
git config --list | grep user              # bash

6.2 Set Global Default Identity

# Set the account you use most often as global default
git config --global user.name  "Raviteja Thota"
git config --global user.email "raviteja1993@gmail.com"

# Check global config file
cat ~/.gitconfig

6.3 Conditional Includes (Auto-Switch by Directory)

The cleanest approach: put all work repos under ~/work/ and personal repos under ~/personal/. Git will automatically use the right identity:

# ~/.gitconfig
[user]
    name  = Raviteja Thota
    email = raviteja1993@gmail.com

[includeIf "gitdir:~/work/"]
    path = ~/.gitconfig-work

[includeIf "gitdir:C:/Users/e135408/work/"]
    path = ~/.gitconfig-work
# ~/.gitconfig-work
[user]
    name  = Raviteja Thota
    email = raviteja.thota@epam.com
✅ Verify the right identity is being used: git log --oneline -1 --format="%an <%ae>"

6.4 SSH Keys — One per Account

Each GitHub account needs its own SSH key. You can't reuse the same public key on two accounts.

# Generate SSH key for personal account
ssh-keygen -t ed25519 -C "raviteja1993@gmail.com" -f ~/.ssh/id_ed25519_personal

# Generate SSH key for work account
ssh-keygen -t ed25519 -C "raviteja.thota@epam.com" -f ~/.ssh/id_ed25519_work

# Add public keys to each GitHub account:
# GitHub.com → Settings → SSH and GPG Keys → New SSH Key
cat ~/.ssh/id_ed25519_personal.pub  # paste into personal account
cat ~/.ssh/id_ed25519_work.pub      # paste into work account

Then configure ~/.ssh/config to use different keys per host alias:

# ~/.ssh/config

# Personal GitHub
Host github-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal
    IdentitiesOnly yes

# Work GitHub
Host github-work
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work
    IdentitiesOnly yes

Now clone using the alias instead of github.com:

# Personal repo clone
git clone git@github-personal:personal-user/my-repo.git

# Work repo clone
git clone git@github-work:work-org/work-repo.git

# Update existing repo remote
git remote set-url origin git@github-work:work-org/existing-repo.git

# Test SSH connection
ssh -T git@github-personal   # → Hi personal-user!
ssh -T git@github-work        # → Hi work-user!

6.5 HTTPS Credentials — Multiple Accounts

HTTPS with Git credential manager supports multiple accounts via the useHttpPath setting:

# Enable per-repository credential storage
git config --global credential.useHttpPath true

# On Windows, GCM (Git Credential Manager) stores per-repo tokens
# You'll be prompted to log in the first time you push from each repo

# Check stored credentials
git credential-manager list
ℹ️ With gh CLI, HTTPS cloning uses the active account's token automatically via the Git credential helper registered during gh auth login. Switch the active account with gh auth switch before cloning new repos.

7. Automation Scripts

7.1 PowerShell Helper Functions

Add these to your $PROFILE (notepad $PROFILE) for quick switching:

# ~/.config/powershell/Microsoft.PowerShell_profile.ps1

function Use-PersonalGitHub {
    $env:GH_TOKEN = (gh auth token --user personal-user 2>$null)
    Write-Host "Switched to personal GitHub account" -ForegroundColor Green
    gh api user --jq '"Active: " + .login'
}

function Use-WorkGitHub {
    $env:GH_TOKEN = (gh auth token --user work-user 2>$null)
    Write-Host "Switched to work GitHub account" -ForegroundColor Cyan
    gh api user --jq '"Active: " + .login'
}

function Get-GitHubUser {
    gh api user --jq '.login + " (" + .name + ")"'
}

# Usage: type these in any terminal
# Use-PersonalGitHub
# Use-WorkGitHub
# Get-GitHubUser

7.2 Bash/Zsh Helper Functions

Add to ~/.bashrc or ~/.zshrc:

# ~/.bashrc or ~/.zshrc

ghpersonal() {
    export GH_TOKEN=$(gh auth token --user personal-user)
    echo "✅ Switched to personal GitHub: $(gh api user --jq .login)"
}

ghwork() {
    export GH_TOKEN=$(gh auth token --user work-user)
    echo "✅ Switched to work GitHub: $(gh api user --jq .login)"
}

ghwho() {
    echo "Active: $(gh api user --jq '.login + " — " + .name')"
}

# Usage:
# ghpersonal  # switch to personal
# ghwork      # switch to work
# ghwho       # check current

7.3 Python .env Pattern for LLM Projects

Keep both tokens available in Python projects without hard-coding:

# .env (never commit this)
PERSONAL_GH_USER=personal-user
WORK_GH_USER=work-user
import subprocess
import os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()

def gh_client(user: str = None) -> OpenAI:
    """Get Copilot Free API client for specified or active GitHub account."""
    cmd = ["gh", "auth", "token"]
    if user:
        cmd += ["--user", user]
    token = subprocess.run(cmd, capture_output=True, text=True).stdout.strip()
    if not token:
        raise RuntimeError(f"No token found for gh user: {user}")
    return OpenAI(
        base_url="https://models.inference.ai.azure.com",
        api_key=token
    )

# Personal account (free Copilot tier)
personal = gh_client(os.getenv("PERSONAL_GH_USER"))

# Work account (Copilot Business/Enterprise)
work = gh_client(os.getenv("WORK_GH_USER"))

# Use whichever is currently active globally
active = gh_client()

resp = active.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Summarize this PR in 3 bullets: ..."}]
)
print(resp.choices[0].message.content)

7.4 Multi-Account API Comparison Script

"""Run the same prompt on both accounts and compare responses."""
import subprocess
from openai import OpenAI
from concurrent.futures import ThreadPoolExecutor

ACCOUNTS = ["personal-user", "work-user"]
PROMPT   = "Explain async/await in Python in 2 sentences."
MODEL    = "gpt-4o-mini"

def ask(user: str) -> tuple[str, str]:
    token = subprocess.run(
        ["gh", "auth", "token", "--user", user],
        capture_output=True, text=True
    ).stdout.strip()
    client = OpenAI(base_url="https://models.inference.ai.azure.com", api_key=token)
    resp = client.chat.completions.create(
        model=MODEL,
        messages=[{"role": "user", "content": PROMPT}]
    )
    return user, resp.choices[0].message.content

with ThreadPoolExecutor() as ex:
    results = list(ex.map(lambda u: ask(u), ACCOUNTS))

for user, answer in results:
    print(f"\n--- {user} ---\n{answer}")

10. Interview Q&A

How does gh CLI store multiple account tokens on Windows?

gh uses the Windows Credential Manager (system keyring) as the primary storage for OAuth tokens — they are encrypted and tied to your Windows user profile. The account metadata (username, protocol, active user) is stored in plaintext in %APPDATA%\GitHub CLI\hosts.yml (or ~/.config/gh/hosts.yml). Running gh auth status shows all registered accounts. Tokens are never stored in plaintext on disk when a keyring is available.

What is the difference between gh auth switch and GH_TOKEN env var?

gh auth switch changes the globally active account — it persists across all terminal sessions by updating hosts.yml. The GH_TOKEN environment variable is a session-scoped override — it only affects the current terminal process and its children, and it takes precedence over the global active account. Unset it to return to the global active account. Use GH_TOKEN for parallel work in different terminal windows.

Can both GitHub accounts use GitHub Copilot Free simultaneously?

Yes — each account has its own Copilot Free quota (2,000 completions/month, 50 chat requests/month as of 2024). You can get a token from each account with gh auth token --user <name> and use them in separate OpenAI client instances pointing to https://models.inference.ai.azure.com. They operate independently — using up one account's quota does not affect the other. This is a legitimate way to increase your free usage for personal experimentation.

How do you ensure Git commits go to the right GitHub account?

There are three levels of control: (1) Global identity — git config --global user.email applies to all repos by default. (2) Per-repo identity — git config user.email (no --global) inside the repo overrides the global setting. (3) Conditional includes — gitconfig [includeIf "gitdir:~/work/"] automatically applies a different config file for all repos under a specific directory. The SSH host alias approach (git@github-work:org/repo.git) handles authentication (which account can push), while user.email handles the commit author identity.

Why can't you use the same SSH key for two GitHub accounts?

GitHub uses SSH public keys as a unique account identifier — each public key can only be registered to one account. If you try to add the same key to a second account, GitHub rejects it with "Key already in use." The solution is to generate a separate key pair per account (ssh-keygen -f ~/.ssh/id_ed25519_work) and configure SSH host aliases in ~/.ssh/config so the correct key is automatically selected when connecting to github-personal vs github-work.

How do you automate account switching in CI/CD pipelines?

In CI/CD, you never use interactive gh auth login. Instead: (1) Create a fine-grained PAT (Personal Access Token) with minimal scopes for the pipeline's account. (2) Store it as a GitHub Actions secret or pipeline secret variable. (3) Set GH_TOKEN=${{ secrets.WORK_GH_TOKEN }} in the workflow environment. (4) All gh commands in that job automatically use that token. For multi-account workflows in a single pipeline, use separate jobs with different secrets, or use gh api with --header "Authorization: Bearer $TOKEN" and pass different tokens per call.

What scopes does a PAT need for full gh CLI functionality?

For full gh CLI functionality: repo (full repo access), read:org (org membership), gist (gist CRUD), workflow (GitHub Actions workflow management), read:packages (container registry), copilot (Copilot API access if needed). For Copilot Free API specifically, the token obtained via gh auth token is an OAuth token, not a PAT — it has broader scopes managed by the OAuth app. For minimal security in automation, use fine-grained PATs scoped to specific repositories and actions.
