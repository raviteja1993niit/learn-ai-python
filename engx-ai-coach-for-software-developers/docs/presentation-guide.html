Q&A Preparation — 20 Expected Questions with Answers

Evaluators will probe technical depth, practical applicability, and your ability to explain concepts at a developer level. Know these cold.

What exactly is the difference between the orchestrator and a sub-agent?
The orchestrator has exactly one job: routing. It receives the user command, classifies the workflow type (A–K in the toggle system), and delegates to the correct sub-agent via run_subagent. It NEVER reads files, never analyses code, never writes reports.

A sub-agent owns one domain completely. SA-1 only does Java scanning. SA-3 only does Confluence lookups. They have different tool sets — SA-3 has mcp-confluence/getPage, SA-1 has grep_search + run_in_terminal.

This separation means: if Confluence's API changes, you only touch SA-3. If you add a new database, you add SA-8 without touching any existing agent.
Why .github/agents instead of LangChain or LangGraph?
Zero infrastructure. No pip install of a 50-dependency framework. No deployment. No runtime server.

.github/agents/ is just Markdown files that GitHub Copilot reads. It runs inside VS Code today, on any developer's machine, with their existing Copilot license.

LangChain/LangGraph are excellent for production pipelines where you need state machines, checkpointing, custom tool definitions in Python, or multi-model routing. But for a developer who wants to ship agentic capabilities in a sprint, .github/agents is the fastest path from idea to running system.

Both are valid. The exam topic says "LangChain, LangGraph, ADK etc." — that "etc." covers Copilot Agents.
How does context isolation work? Does each sub-agent see the full conversation?
No — that's a key design principle. The orchestrator passes a sealed context packet to each sub-agent. This packet contains only what that agent needs: the toggle name, the relevant CSV paths, the Confluence base URL, etc.

Sub-agents do NOT see the user's full conversation history. They do NOT see each other's outputs directly. They return a compact status report to the orchestrator, which aggregates and presents the final answer.

This is why the system can scale to analysing 100+ toggles — each invocation has a bounded, predictable context size regardless of conversation length.
What is the run_subagent tool? Is it a standard tool?
run_subagent is a GitHub Copilot built-in tool available in agent mode. It spawns a sub-agent by name, passes it a context string, and returns the sub-agent's response.

In the agent Markdown file you declare it in the tools: array. The orchestrator calls it like: run_subagent("toggle-analysis-engine", context_packet).

Only the orchestrator has this tool. Sub-agents (SA-1 to SA-7) do NOT have run_subagent in their tools array — this enforces the hierarchy and prevents sub-agents from spawning uncontrolled agent chains.
What is MCP? How does the Confluence integration actually work?
MCP (Model Context Protocol) is an open protocol by Anthropic that lets LLMs call external tools in a standardised way. You define a server that exposes tools, and the agent calls them as if they were local functions.

For Confluence: the MCP server wraps the Confluence REST API. SA-3 calls mcp-confluence/getPage, searchContent, and getPageBatch. The MCP server authenticates, fetches, and returns the page content as structured JSON. SA-3 then extracts the fields it needs (URL, summary, is_s2a, team, apps).

4-tier search strategy: exact toggle name → fuzzy name → team-scoped search → batch fetch. Each tier runs only if the previous one returns no results.
Why does vector-rag-app-v1's requirements.txt mention sentence-transformers and chromadb but the code uses TF-IDF?
Great catch — that's a real artifact of iterative development. The requirements.txt was written at the start of the project when the plan was to use sentence-transformers + ChromaDB for semantic embedding.

During implementation, I discovered that TF-IDF + LSA (TruncatedSVD) from scikit-learn gave similar retrieval quality for structured documents at zero GPU cost and with 2-second indexing time vs 30+ seconds for embedding models.

The code evolved; the requirements file didn't get updated. This is actually a teaching moment for the RAG evolution story — even the first version moved away from the "full vector DB" approach once the practical trade-offs became clear.
How does ask.py handle large documents that exceed the LLM's context window?
convert.py + ask.py are optimised for structured documents up to ~100 pages, which fits within modern model context windows (128k tokens for Claude Sonnet, ~100k for GPT-4o).

For larger documents, summarize.py pre-processes each section independently in concurrent batches of 5, truncating sections over 6000 characters before summarising. This produces a .summary.json which is much smaller and can be fed to ask.py.

For truly massive corpora (thousands of documents), V1's chunking + vector search is the right choice. The decision rule is: "Can you fit the full doc in context? → V3. Large corpus → V1."
How does the transcript notes generator work? Is it real-time?
It uses two modes:

1. File transcription: Upload a pre-recorded audio/video file. audio_transcriber.py runs faster-whisper locally (cached WhisperModel), applies VAD (voice activity detection) to skip silence, and returns joined transcript text.

2. Live recording: system_audio_recorder.py captures Windows system audio via WASAPI loopback using pyaudiowpatch. A background thread records PCM frames, downmixes to mono, resamples to 16kHz, and writes a WAV header. This is then passed to faster-whisper.

The agent (transcript-guide.agent.md) then analyses the narrator's style, chunks the transcript in 5-line loops, and builds a white-paper style guide. Entirely local — no audio sent to cloud services.
How would you add a new agent to the toggle system without breaking existing ones?
Three steps:
  1. Create .github/agents/toggle-newagent.agent.md with its name, tools, and system prompt.
  2. Register it in .github/agent-registry.yml — add the agent name, workflow trigger, tools, and context isolation budget.
  3. Add a routing rule to the orchestrator's system prompt: "If workflow type X, delegate to toggle-newagent."
Existing agents are untouched. The orchestrator is the only file that references agent names — and even then only via workflow type classification, not hard-coded logic. This is the composability principle in action.
What is S2A field notation? Why does it matter?
S2A (Service-to-API) is a specific field annotation pattern used in the MPGS codebase to trace which API fields in the payment gateway specification map to which Java service fields.

When SA-1 analyses a toggle, it doesn't just find ON/OFF branches — it also identifies which S2A-annotated fields are inside each branch. This tells the team: "If we flip this toggle OFF, we lose visibility into payment field X in API spec section Y."

This is domain-specific knowledge baked into the agent's system prompt. It's an example of how agents are more powerful than generic LLM prompts: you can embed expert domain knowledge that stays consistent across all analyses.
How is this different from just using GitHub Copilot normally?
Normal GitHub Copilot = one model, one context, one response. You write a prompt, it answers, conversation ends or continues in one thread.

Multi-agent Copilot with .github/agents = persistent specialised agents that remember their role, have scoped tool access, and can be invoked by name. The orchestrator can delegate to SA-1 without SA-1 knowing about SA-3's conversation.

Practically: you can type @toggle-orchestrator analyze toggle MPGS_3DS_V2 and the orchestrator will spawn SA-1, SA-3, and SA-2 in sequence, using shared state from the workspace, without you writing a single multi-step prompt. It's a persistent workflow, not a one-shot response.
Why BM25 instead of semantic embeddings in V2? Isn't semantic search better?
Semantic search (embeddings) is better for meaning-based retrieval when the query and document use different words. For example: query "payment failure" matching document text "transaction declined".

But for technical specification documents (acquirer specs, API docs, legal text), exact terminology matters. The spec says "ISO 8583 field 39" — you want to match that exactly, not semantically approximate it. BM25 excels at this.

Additionally: zero GPU, zero model download, 5-second setup, deterministic results (same query = same pages every time), and you can visualise which pages were matched via the score chart. For structured business documents, BM25 is often the pragmatic right answer.
What happens if the LLM hallucinates a toggle location in SA-1?
The system has three safeguards:

1. 3-tier smart lookup: SA-1 first checks the toggle registry (a CSV of known locations), then does a live grep_search in the actual codebase, then escalates to SA-6 (manual lookup agent) if tiers 1 and 2 fail. SA-6 does a deep filesystem search and only writes to registry after user confirmation.

2. POST-hook validation: After every analysis, SA-1 runs a stuck check. If confidence is low, it flags the orchestrator rather than writing a bad result.

3. NOT_FOUND protocol: If no location is found after all tiers, the orchestrator returns a structured NOT_FOUND response to the user rather than guessing. Explicit uncertainty is better than silent hallucination.
Could this work with Claude Code or other coding agents?
Yes — and the architecture is similar. Claude Code uses a tool-calling loop where the agent can use bash, read/write files, and spawn sub-processes. You could replicate the orchestrator + sub-agent pattern by giving each "agent" a separate system prompt and having the main agent call them via subprocess or API.

The difference with .github/agents is it's native to VS Code and GitHub Copilot — zero extra tooling for developers already using Copilot. Claude Code is more powerful for long-running autonomous tasks but requires separate access.

The orchestrator pattern itself (one router, N specialists, shared state) is framework-agnostic. LangGraph would express this as a directed graph of nodes. ADK as an agent graph. .github/agents as Markdown files. Same mental model, different syntax.
How would you scale this to 10 developers on the same team?
The .github/agents/ folder lives in the repository. When a developer clones the repo, they get all agent definitions immediately.

Shared state (registry CSV, analysis CSVs, Confluence results) live in the repo's data/ folder — committed and versioned. SA-5 (workspace-manager) handles onboarding new developers: /toggle-onboard-repo adds a new repo to config.

For team-wide deployment: commit the .github/agents/, knowledge/, and data/ folders. Each developer runs their own Copilot instance but shares the same agent definitions and accumulated knowledge (registry, reports).

Recommendation: keep data/ gitignored for sensitive analysis results, but keep knowledge/config.yml and agent definitions versioned.
What is the convert.py JSON tree format? Why a tree?
convert.py produces a nested JSON structure: { "title": "...", "sections": [ { "title": "...", "content": "...", "tables": [...] } ] }

The tree preserves the document's logical structure — headings, subsections, tables — rather than flattening it into page numbers or character offsets.

For PDF: uses TOC/bookmarks if present, falls back to heuristic heading detection (large bold text at start of paragraphs), falls back to 5-page batches. For DOCX: uses Heading 1/2/3 styles; extracts tables into structured rows. For XLSX: each sheet becomes a section.

Why tree? Because ask.py's build_context() rebuilds the document with section headers preserved, so the LLM can say "In Section 4.2 (Authentication Flow), the retry limit is 3" — not just "on page 47" which has no meaning after chunking.
How does the token authentication work for the LLM calls?
All apps use GitHub Models / GitHub Copilot's OpenAI-compatible API. Token resolution order (from src/llm.py):

1. Command-line argument --token
2. Environment variable GITHUB_TOKEN
3. gh auth token — runs the GitHub CLI to get the logged-in user's token

On Windows, the LLM client tries WinHTTP via PowerShell first (for enterprise proxy compatibility), then falls back to httpx.

The base URL is https://models.inference.ai.azure.com — GitHub's model inference endpoint. Model used: gpt-4o or claude-sonnet-4-5 depending on the app config. Live model discovery via /models endpoint is implemented in the transcript app.
Can this pattern be used outside of VS Code? E.g., in a CI/CD pipeline?
The .github/agents format is VS Code / GitHub Copilot specific. For CI/CD you'd need a different approach:

Option 1: Port the agent logic to Python scripts with an LLM API. The orchestrator becomes a function that dispatches to specialist functions. SA-1 becomes java_analyzer.py. Same pattern, different runtime.

Option 2: Use GitHub's Copilot Extensions API (currently in beta) to expose agents as API endpoints that can be called from Actions.

Option 3: LangGraph for production pipelines — define the same graph topology (orchestrator node → specialist nodes) with checkpoint support for long-running CI jobs.

The principle is CI/CD compatible — the Python scripts in tools/ (build-toggle-registry-lookup.ps1, toggle-search-utility.ps1) already run standalone.
Why did you build 6 apps instead of one comprehensive system?
Intentional — each app isolates one architectural decision so the evolution is teachable:

• vector-rag-app-v1: baseline — "what most tutorials show you"
• vector-less-rag-app: first simplification — chunk-based BM25
• vector-less-rag-app-v2: refined BM25 — page-based, FastAPI, score viz
• vector-less-rag-agentic: the insight — LLM doesn't need a retriever
• agentic-ai-toggle-management: full multi-agent system in a real domain
• transcript-notes-generator: shows the pattern applied to audio/transcription

If I had built one monolith, the teaching story would be invisible. The six separate repos make the evolution of thinking explicit and demonstrable.
What would you do differently if you rebuilt the toggle system today?
Three things:

1. Add structured output validation: Each sub-agent should return a typed JSON schema response, not free-form text. The orchestrator would validate the response shape before proceeding. This prevents silent failures.

2. Event-driven triggers: Instead of slash commands, hook into git push events. When a developer commits a change to a Java file containing a toggle, SA-1 auto-triggers analysis and posts a PR comment with the impact report.

3. Agent telemetry: Track token usage and latency per agent per invocation. The knowledge/config.yml context isolation budgets are static — they should adapt based on observed p95 token usage over time.
