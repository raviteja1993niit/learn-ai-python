Comprehensive Reference Guide  v1.1.0Beginner Friendly

🗺️ What This App Does

A Streamlit web app that converts raw text transcripts (and audio recordings) into well-structured Markdown guides using GitHub Copilot AI models — completely free, with no paid API keys required.

Analogy: You paste a messy YouTube transcript and get back a polished study guide — with headings, tables, code blocks, and key takeaways — automatically.

Key highlights:

  • No paid API keys needed — uses your GitHub Copilot subscription via gh auth token
  • Two tabs: 📝 Transcript → Markdown and 🎙️ Audio → Markdown
  • Disk cache for repeated transcript chunks — no wasted API calls
  • Handles transcripts of any length via chunked pipeline with compression
TabInputWhat HappensOutput
📝 Transcript → MarkdownPaste text or upload .txtAI chunks + structures the content.md file download
🎙️ Audio → MarkdownRecord system audio, mic, or upload audio fileWhisper transcribes locally → AI structures.md file download

🔧 Prerequisites

Three things must be in place before you can run the app:

  1. Python 3.10+ — Check with python --version. Download from python.org if needed.
  2. GitHub CLI (gh) authenticated with Copilot — Install from cli.github.com, then run gh auth login.
  3. Python packages from requirements.txt — Installed in the Installation step below.
⚠️ Windows Audio Capture Note
System audio capture requires pyaudiowpatch and a WASAPI loopback-compatible audio driver. Most Intel, Realtek, and Conexant drivers work. AMD audio drivers sometimes don't expose a loopback device. If you can't capture system audio, use the Upload option instead.

Why gh auth token?

Instead of paying for an API key, the app uses your existing GitHub Copilot subscription. The GitHub CLI stores your authentication token securely in your OS credential store. The app reads it with one command:

import subprocess
token = subprocess.run(["gh", "auth", "token"], capture_output=True, text=True).stdout.strip()

This is identical to how Copilot-powered editors authenticate — the same subscription, the same models, zero extra cost.

⬇️ Installation

  1. Clone or download this repository to your machine.
  2. Open a terminal in the project folder — the one containing app.py.
  3. Install Python packages: python -m pip install -r requirements.txt
  4. Authenticate with GitHub CLI: gh auth login — follow the prompts to log in via browser.
  5. Start the app: streamlit run app.py
  6. Open your browser at http://localhost:8501
ℹ️ First-Time Audio Use
The first time you use the Audio tab, Whisper downloads the base model (~145 MB). It is cached at ~/.cache/huggingface/hub/ — subsequent runs are instant. No internet connection required after the first download.

📝 Transcript → Markdown Flow

The pipeline has two branches depending on the transcript length. Short transcripts go straight to synthesis; long ones are chunked, cached, and locally compressed before a final synthesis call.

Short Transcript (under 8,000 chars)

User pastes text │ ▼ _normalise() ← strips timestamps, speaker labels, collapses whitespace │ ▼ _single_pass() ← one LLM call: full text + _SYNTHESIS_SYSTEM prompt │ ▼ ✅ Structured Markdown

Long Transcript (over 8,000 chars)

User pastes text │ ▼ _normalise() ← strips timestamps, speaker labels │ ▼ _split_chunks() ← sentence-boundary splits into ~8,000-char pieces │ ▼ [for each chunk] Check disk cache? ├── HIT → load notes instantly (no API call) └── MISS → _extract_notes() via LLM → save to cache │ + adaptive 2s pause between calls │ ▼ _dedup_notes() ← removes exact duplicate bullet lines (15–30% reduction) │ ▼ [if notes > 16,000 chars] _local_compress_notes() ← Python only, zero API calls → dedup + prioritise + trim _compress_notes_batched() ← LLM fallback only if local compression isn't enough │ ▼ _synthesise_from_notes() ← one final LLM call: structured Markdown │ ▼ ✅ Structured Markdown

app.py

🖥️

app.py

Main entry point  ·  ~504 lines  ·  Streamlit UI code only — no business logic

app.py is the "manager" — it renders the browser interface and delegates all real work to the other modules. It never directly calls the Copilot API; it calls transcript_processor.process() and audio_transcriber.transcribe_bytes().

Sidebar & Settings

The sidebar is rendered first on every Streamlit rerun. It performs these steps in order:

  1. Token check — calls get_gh_token(). If no token is found: shows a red error message and calls st.stop() to halt rendering.
  2. Model list — calls get_available_models(token) which fetches the live model list from GitHub's API once per session. Speed hint labels (⚡ fast, ⚖️ balanced, 🧠 powerful) are appended.
  3. Model persistence fix — reads st.session_state["selected_model"] to preserve the user's last choice across reruns.
  4. Temperature slider — 0.0 (focused, deterministic) to 1.0 (creative, varied). Default 0.3.
  5. Cache controls — a checkbox to enable/disable the disk cache, stats showing entry count + disk usage, and a Clear button.
  6. Synthesis model override — an advanced expander allowing a different (usually more powerful) model for the final synthesis step only.
  7. Whisper model selector — tiny / base / small / medium, with speed and accuracy hints.

Why Model Selection Used to Reset — and How It's Fixed

Streamlit reruns the entire app.py on every button click. Without a fix, the selectbox recalculates default_idx from DEFAULT_MODEL every time — resetting the user's choice.

# Before rendering the selectbox, read the previously saved choice:
saved_model = st.session_state.get("selected_model", copilot_llm.DEFAULT_MODEL)
default_idx = model_names.index(saved_model) if saved_model in model_names else 0

model_choice = st.selectbox("Model", model_names, index=default_idx)

# After the selectbox renders, save the current choice:
st.session_state["selected_model"] = model_choice

src/llm.py

GitHub Copilot API wrapper  ·  ~353 lines

The "messenger" — takes your chat messages and delivers them to GitHub's servers, handling all network complexity: token retrieval, model listing, platform differences (Windows vs Linux), and retry logic.

Token Resolution

get_gh_token() checks two places in order:

def get_gh_token() -> str:
    # 1. Check environment variable (useful in CI/CD or Docker)
    if token := os.environ.get("GITHUB_TOKEN", ""):
        return token
    # 2. Ask the GitHub CLI tool (standard desktop usage)
    result = subprocess.run(
        ["gh", "auth", "token"],
        capture_output=True, text=True, timeout=10
    )
    return result.stdout.strip()

Model Fetching

Available models are fetched live from https://api.githubcopilot.com/models once per Python process (module-level cache).

COPILOT_MODELS: list[str] = []  # module-level cache — empty until first call

def get_available_models(token: str) -> list[str]:
    if COPILOT_MODELS:           # already fetched — return immediately
        return COPILOT_MODELS
    # ... fetch from API, deduplicate, sort by capability, cache in COPILOT_MODELS ...
    return COPILOT_MODELS

Post-fetch processing:

  • _dedupe_by_version(): groups model aliases by their version field, keeps the shortest clean ID (e.g., gpt-5-mini instead of gpt-5-mini-2025-01-15).
  • Filters out text-embedding-* (search models, not chat) and *-1m (internal/restricted).
  • Models are sorted: fast first, then balanced, then powerful, then code-specialist.

Chat API Call

def chat(
    messages: list[dict[str, str]],  # [{"role": "system", "content": "..."}, ...]
    model: str = DEFAULT_MODEL,      # "gpt-5-mini" by default
    temperature: float = 0.3,        # creativity level (0 = focused, 1 = creative)
    max_retries: int = 4,            # retry on transient failures
) -> str:
    # On Windows:   use PowerShell's Invoke-RestMethod
    #               (works behind corporate SSL-inspecting proxies)
    # On Linux/Mac: use httpx
    #               (pure Python, no subprocess overhead)
    ...

Why PowerShell on Windows?

Corporate environments often deploy SSL inspection proxies. Python's ssl module uses its own certificate bundle and doesn't trust the corporate CA certificate — causing SSLError. PowerShell's Invoke-RestMethod uses the Windows certificate store, which automatically includes the corporate CA. Result: it works where httpx fails.

Temp File Trick

The request body is written to a temporary .json file rather than passed as an inline string. This avoids PowerShell's apostrophe/quote escaping nightmare when transcript text contains single quotes.

Normalise & Chunk

_normalise() cleans the raw transcript before any AI processing:

def _normalise(text: str) -> str:
    # Remove YouTube timestamps like "0:32" or "1:23:45"
    text = re.sub(r'\b\d{1,2}:\d{2}(?::\d{2})?\b', '', text)
    # Remove "Speaker 1:" or "John:" or "[Host]:" labels
    text = re.sub(r'^\[?[A-Za-z][A-Za-z0-9 ]{0,30}\]?\s*:\s*', '', text, flags=re.MULTILINE)
    # Collapse multiple spaces, tabs, and newlines into single spaces
    text = re.sub(r'\s+', ' ', text)
    return text.strip()

_split_chunks() splits the normalised text at sentence boundaries into pieces of approximately _CHUNK_SIZE characters (default 15,000 — halves API calls vs the old 8,000).

ℹ️ Why 8,000 Characters?
8,000 chars ≈ 2,000 tokens — fits comfortably within the model's context window while keeping API calls manageable. The old value was 3,000 characters, which produced 99 chunks for a 1-hour transcript. At 8,000, that's ~37 chunks — a 3× speedup with no quality loss.

Note Extraction

_extract_notes() sends each chunk to the LLM with the _NOTES_SYSTEM prompt, which instructs the AI to extract bullet-point notes covering concepts, analogies, code, warnings, quotes, and comparisons.

Example output for a Docker chunk:

- Docker containers share the host OS kernel — unlike VMs which each have their own OS
- Analogy: containers are like shipping containers — standard interface, any contents
- docker run -d -p 80:80 nginx — runs Nginx in background, maps port 80
- Warning: data inside a container is lost when it stops — use volumes for persistence
- containers start in ~100ms; VMs take minutes

Adaptive Delay Logic

# If a call took > 30 seconds (likely hit a rate-limit retry), back off:
if call_elapsed > 30:
    delay = min(delay * 1.5, _INTER_CHUNK_DELAY_MAX)
# If 3 consecutive fast calls, speed up again:
elif fast_streak >= 3 and delay > _INTER_CHUNK_DELAY_BASE:
    delay = max(delay * 0.75, _INTER_CHUNK_DELAY_BASE)

_dedup_notes() removes exact duplicate bullet lines across all chunks. Typical reduction: 15–30% (introductory topics often re-explained multiple times in long videos).

Compression

When combined notes exceed _MAX_SYNTHESIS_CHARS (80,000 chars), local Python compression runs first — zero API calls. _local_compress_notes() deduplicates near-identical bullets, truncates verbose lines, and proportionally trims with priority given to code, warnings, and quotes. Only if notes still exceed 80k after local compression does it fall back to _compress_notes_batched() (LLM calls — rare).

all_notes (16 chunk-notes, e.g. 101k chars) │ ▼ [_local_compress_notes() — Python only, ~instant] 1. Deduplicate near-identical bullets across all chunks 2. Prioritise: code (4) > warnings (3) > quotes (2) > bullets (1) 3. Truncate lines > 200 chars 4. Budget-trim each chunk proportionally to hit target │ ▼ [result e.g. 45k chars — under 80k limit] sent to _synthesise_from_notes() [LLM fallback — only if >80k after local compression] Batch 1 (chunks 1–5) → LLM compress → summary_1 Batch 2 (chunks 6–10) → LLM compress → summary_2 …

Final Synthesis

_synthesise_from_notes() sends the (optionally compressed) notes to the LLM with the _SYNTHESIS_SYSTEM prompt — one single call that produces the complete Markdown document.

Rules the AI follows from _SYNTHESIS_SYSTEM:

  • Exactly ONE H1: # {Topic} — A Complete Guide
  • H2 major sections, H3 sub-topics, H4 details — never skip levels
  • Tables only where the source compares or lists multiple items
  • ASCII diagrams only where flows or architectures are described
  • Code blocks only where real code appeared in the transcript
  • Key Takeaways section at the end
  • No invented content — only what was in the transcript

How transcribe_bytes() Works

  1. Load model — first call downloads weights to ~/.cache/huggingface/hub/; subsequent calls reuse the Python object stored in _model_cache.
  2. Write to temp file — faster-whisper requires a file path (not raw bytes), so bytes are written to a temporary file.
  3. Run inference — beam_size=5 for quality + VAD filter to skip silent gaps.
  4. Collect segments — all segment texts are joined with spaces to form the full transcript.
  5. Cleanup — temp file is deleted; if the result is empty, a RuntimeError is raised so the UI can show a helpful message.

VAD Filter

The Voice Activity Detection filter skips silent gaps ≥ 500 ms. This is especially useful for system audio recordings with pauses between video sections — it prevents Whisper from hallucinating content during silence.

Language Parameter

Default is None = auto-detect from the first 30 seconds. Setting an explicit code like "en" is faster (skips detection) and more accurate (Whisper doesn't waste probability mass on other languages).

Audio Processing in _to_wav_bytes()

  1. Parse samples — raw bytes are int16; NumPy reshapes to [sample, channel] array.
  2. Downmix to mono — stereo averaged: mono = audio.mean(axis=1)
  3. Resample to 16 kHz — SciPy resample_poly converts 44.1 kHz → 16 kHz (Whisper requires 16 kHz).
  4. Write WAV header — 44-byte RIFF/WAV header prepended; result is a valid .wav file in memory.
⚠️ No Loopback Devices Found?
This can happen with AMD audio drivers (less reliable loopback support), Bluetooth-only audio output, or virtual machines. Workaround: use the Upload option instead, or enable Stereo Mix in Windows Sound Settings → Recording tab → right-click empty area → "Show disabled devices".

💾 Disk Cache

Processing a 1-hour transcript = ~37 API calls. Re-running the same transcript would waste 10 minutes and repeat all those calls. The disk cache prevents this by storing each chunk's extracted notes on disk after the first run.

Analogy: Like a homework answer sheet — the first time you solve a problem, write down the answer. Next time you see the same problem, just look it up.

Cache Key Design

# SHA-256 hash of everything that affects the output
raw = f"{chunk_text}|{model}|{temperature:.4f}|{first_64_chars_of_prompt}|v{PIPELINE_VERSION}"
key = hashlib.sha256(raw.encode()).hexdigest()
# Result: 64-char hex like "a3f8b2c1d4e5f6a7b8c9..."

Any change to any of these inputs produces a completely different hash → cache miss → fresh API call:

  • The chunk text itself (different transcript = different notes)
  • The model name (gpt-5-mini vs claude-sonnet = different notes)
  • The temperature value
  • The extraction prompt (bump _PIPELINE_VERSION to invalidate all existing cache)

♻️ Run Snapshots & Resume

When processing a long transcript (e.g. 29 chunks), a 403 rate-limit error on chunk 15 used to discard all progress. The next run would start over — re-extracting all 29 chunks (14 already in cache, 15 new API calls). Run snapshots fix this permanently.

Additionally, when a 403 occurs, the UI now shows a countdown during the retry wait ("⏳ Rate limited (403) — retry 1/2 in 55s…") and a clear snapshot-saved message if all retries are exhausted ("💾 Snapshot saved — wait ~60s, click ▶ Process to resume from chunk 15").

How It Works

After every successful event in the pipeline, a small JSON checkpoint is written atomically to output/cache/runs/{run_key}.json. If the process dies at any point, the next run detects the snapshot and jumps directly to the right stage:

Snapshot StageSaved WhenOn Resume
"extracting" After every chunk extraction (API call or cache hit) Reloads notes for completed chunks from chunk cache; resumes extraction loop from next chunk
"compressing" After every compression batch Skips extraction entirely; resumes compression from the next unfinished batch
"synthesising" When all extraction + compression is done Skips everything; runs only the final synthesis call (~20s)

Snapshot File Structure

Each snapshot is a plain JSON file. Here is what each stage stores:

// Stage: "extracting" — saved after each chunk
{
  "schema": 1,
  "pipeline_version": "1",
  "stage": "extracting",
  "total_chunks": 29,
  "last_completed_chunk": 14,
  "chunk_keys": ["a3f8b2...", "c91d40...", ...]   // SHA-256 keys → verified in chunk cache
}

// Stage: "compressing" — saved after each compression batch
{
  "stage": "compressing",
  "total_chunks": 29,
  "all_notes": ["=== CHUNK 1 NOTES ===\n...", ...],
  "last_completed_batch": 2,
  "completed_batches": ["=== COMPRESSED BATCH 1 ===\n...", ...]
}

// Stage: "synthesising" — saved once, before the final LLM call
{
  "stage": "synthesising",
  "total_chunks": 29,
  "final_notes": "... compressed notes ready for synthesis ..."
}

Run Key Design

The snapshot filename is a SHA-256 of the complete normalised transcript + model + temperature + pipeline version. This guarantees:

  • Different transcript → different key → fresh run (no cross-contamination)
  • Different model or temperature → different key → fresh run
  • Bumping _PIPELINE_VERSION → different key → all snapshots auto-invalidated
  • Changing only the synthesis model override → same extraction snapshot reused ✅

Atomic Writes

Snapshots are written to a .tmp file first, then renamed to .json. If the process is killed mid-write, the original .json snapshot stays intact — never leaving behind corrupt JSON that would block a future resume.

🤖 Model Selection

The app fetches available models live from the Copilot API. Speed hint labels are added to help you pick the right model for your situation.

LabelModelsSpeed (synthesis)QualityBest For
⚡ fastgpt-5-mini, gpt-5.4-mini, claude-haiku-4.5~20–40sGoodAll phases — default choice
⚖️ balancedgpt-4.1, gpt-5.2, claude-sonnet-4.6~60–120sBetterSynthesis override only
🧠 powerfulgpt-5.4, gpt-5.5, claude-opus-4.7~120–300sBestAvoid — hits 300s timeout on large guides
💻 codegpt-5.2-codex, gpt-5.3-codex~30–60sBest for codeProgramming tutorials, synthesis override
⚠️ Bigger ≠ Faster

Switching to claude-sonnet-4.6 or claude-opus-4.7 does not reduce 403 rate limits. The 403s come from the Copilot API tier (free = ~5–6 calls per 5 min window), not from the model. Larger models are slower per call and use more quota — making rate limits worse. Use gpt-5-mini for extraction (many calls) and optionally override to a better model for the final synthesis step only (1 call, quality matters most there).

✅ Best Strategy for Long Transcripts
Use gpt-5-mini for chunk extraction (many calls — speed matters) and gpt-5.5 as the Synthesis model override (one final call — quality matters). This gives the fastest processing with the best final output quality.

⏱️ Estimated time: ~2m 54s Rough time forecast for the remaining API calls. Can vary based on how fast the API responds. ✅ Just an estimate
📝 Single-pass processing (short transcript)… Transcript is short enough to process in one API call — no chunking needed. ✅ Fast path

Per-Chunk Processing Lines

Log LineWhat It MeansIs This Good?

🔍 Extracting chunk 9/16… (elapsed 242s · ~160s remaining · delay=10s) Sending section 9 to the AI. elapsed = total time so far. remaining = estimated time left. delay = the pause it will add after this call. ✅ Normal — working

These appear when the GitHub Copilot free API temporarily blocks a request. The app handles them automatically — you do not need to do anything while you see these.

🔍 Troubleshooting Guide

Processing Is Very Slow

  • Switch to gpt-5-mini in the sidebar — fastest model, still good quality
  • Make sure the Use cache checkbox is ON — cached chunks skip API calls entirely
  • If you've processed this transcript before, the sidebar cache stats should show entries
  • If hitting 403s repeatedly, increase _INTER_CHUNK_DELAY_BASE to 4.0 in transcript_processor.py

Output Quality Is Poor

  • Try gpt-5.4 or gpt-5.5 as the Synthesis model override — one more powerful call for the final output
  • Slightly increase temperature to 0.4–0.5 for more expressive prose
  • If the transcript quality is low (e.g., auto-generated captions), try cleaning it manually before pasting

System Audio Not Recording Video Sound

  • Make sure audio is actually playing when you click Start Recording
  • Try a different loopback device in the dropdown — there may be multiple
  • Open Windows Sound Settings → Recording tab → right-click empty area → "Show Disabled Devices" → enable Stereo Mix
  • Intel SoundWire loopback is the most reliable on modern Intel laptops

Whisper Transcription Is Inaccurate

  • Switch from base → small model for better accuracy with accents or technical terms
  • Set language explicitly in the dropdown (auto-detect sometimes fails for short clips)
  • System audio loopback recordings are almost always better quality than microphone recordings
