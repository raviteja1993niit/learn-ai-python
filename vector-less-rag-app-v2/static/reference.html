Corporate SSL Proxy — Root Cause & Fix
Why Python httpx fails and PowerShell works
⚠️
Corporate SSL Inspection (MITM Proxy) Many corporate networks use a transparent HTTPS proxy that re-signs TLS certificates with a corporate CA. Python's httpx uses the certifi bundle which doesn't include the corporate CA → intermittent 403/SSL errors.
❌ Python httpx — Fails
🔴 Uses certifi CA bundle
🔴 Doesn't trust corporate CA
🔴 SSL handshake rejected by proxy
🔴 ~40% failure even with verify=False
🔴 Rapid calls trigger rate-limiting
✅ PowerShell WinHTTP — Works
🟢 Uses Windows system certificate store
🟢 Corporate CA already trusted by Windows
🟢 No SSL rejections from proxy
🟢 Invoke-RestMethod fully reliable
🟢 Exponential backoff handles rate limits

PageIndex & BM25 — How It Works
The core retrieval mechanism — no vectors, no GPU
Traditional RAG (Chunking)
🔴 Split every page into 512-char chunks
🔴 Tables split across chunk boundaries
🔴 Embed each chunk (GPU or API cost)
🔴 Store in VectorDB (FAISS, Chroma, Pinecone)
🔴 Semantic search (approximate)
🔴 Tune chunk_size, overlap, threshold
PageIndex (This App)
🟢 Each document unit = one Page (natural)
🟢 Tables preserved intact on their page
🟢 No embeddings — BM25 is instant
🟢 In-memory Python dict — no external DB
🟢 Exact keyword search (deterministic)
🟢 Only tune Top-K

BM25 Scoring Formula

📐
BM25(D, Q) = Σ IDF(qᵢ) × [ f(qᵢ,D) × (k₁+1) ] / [ f(qᵢ,D) + k₁ × (1 - b + b × |D|/avgdl) ]
Where f(qᵢ,D) = term frequency in doc, |D| = doc length, avgdl = avg doc length, k₁=1.5, b=0.75. A page with query term appearing in the right density wins — not just raw frequency. 
# src/page_index.py — core search
def search(self, query: str, top_k: int = 5):
    tokens = _tokenize(query)          # lowercase alphanumeric
    scores = self._bm25.get_scores(tokens)  # BM25Okapi over all pages
    ranked = sorted(zip(self.pages, scores), key=lambda x: x[1], reverse=True)
    results = [(p, float(s)) for p, s in ranked[:top_k] if s > 0]
    if not results:  # fallback: query terms not in corpus
        results = [(p, 0.0) for p in self.pages[:top_k]]
    return results

Multi-Query Retrieval
Why single BM25 misses acronyms and the fix
⚠️
The TVR Problem Searching "TVR" with a single BM25 query ranks pages where TVR appears many times (changelog, cross-references) higher than the actual definition page — which might have TVR only once in a table row. Multi-query retrieval fixes this. 
# src/rag.py — multi-query strategy
def _multi_query_search(self, question: str):
    combined = {}

    # 1. Full question search (2× top_k to cast wide net)
    for page, score in self.index.search(question, top_k=self.top_k * 2):
        combined[page.page_num] = (page, score)

    # 2. Per-token search (especially good for short acronyms: TVR, EMV, ICC)
    STOP = {"the", "and", "for", "get", ...}
    tokens = [t for t in re.findall(r"[A-Za-z0-9]+", question)
              if len(t) >= 2 and t.lower() not in STOP]

    for token in tokens:
        for page, score in self.index.search(token, top_k=self.top_k):
            # Keep highest score per page
            if page.page_num not in combined or score > combined[page.page_num][1]:
                combined[page.page_num] = (page, score)

    # 3. Merge, re-rank by score, return top_k
    return sorted(combined.values(), key=lambda x: x[1], reverse=True)[:self.top_k]

Multi-Turn Chat History
How follow-up questions retain context
# Each session stores a growing list of turns:
chat_history = [
    {"role": "user",      "content": "What is TVR?"},
    {"role": "assistant", "content": "TVR (Terminal Verification Results) is..."},
    {"role": "user",      "content": "What is the field length?"},  ← follow-up
    ...
]

# Each new question includes the last 6 turns of history (3 exchanges)
# System prompt + history + new context + new question → LLM
💡
Follow-up questions work even without document keywords "What is the field length?" refers back to "TVR" from the prior turn because the full history is sent to the LLM on every request.

REST API Reference
All FastAPI endpoints exposed by server.py
GET / Serves the main SPA (index.html)
Returns the full ChatGPT-style chat UI. No parameters.
GET /api/models List all available models by provider
Returns: {"models": {"GitHub Copilot": [...], "GitHub Models": [...]}}
GET /api/token Auto-detect GitHub token from gh CLI
Returns: {"token": "gho_xxxx"} or {"token": ""} if not found.
POST /api/upload Upload and index a document
Form fields: session_id, file (binary), model, provider, github_token, top_k
Returns: {"ok": true, "filename": "...", "stats": {"total_pages": N, "total_words": N, ...}}
POST /api/chat Ask a question about the loaded document
Body: {"session_id": "uuid", "question": "What is TVR?"}
Returns: {"answer": "...", "sources": [{"page_num": N, "title": "...", "score": 0.95, ...}]}
GET /api/pages?session_id=&search= Browse all pages or search by relevance
Returns: {"pages": [{page_num, title, word_count, source_type, content, score}], "stats": {...}}
POST /api/reset Clear session (New Chat)
Body: {"session_id": "uuid"}. Removes pipeline and chat history from server memory.
POST /api/clear_chat Clear chat history only (keep document indexed)
Body: {"session_id": "uuid"}. Clears conversation but keeps PageIndex in memory.

Top-K Tuning Guide
The single most important parameter — how to set it
Document TypeRecommended Top-KWhy
Short reports (1–20 pages)3–4Few pages, high precision better
Business docs, manuals (20–100 pages)6–8Good balance of coverage and context
Technical specs (100–500 pages)8–12Many cross-references, definitions scattered
Large PDFs (500+ pages)10–15Cast wide net; LLM context is 24K chars
Acronym / field lookup10+Definition page may rank lower in BM25
💡
Context budget is 24,000 characters Each page averages ~1,500 chars. At Top-K=10, you get ~10 full pages in context before truncation. The LLM sees: system prompt + chat history (last 6 turns) + retrieved pages + current question.
